CH 01
Introduction to Software Engineering
What is SE · Products · Ethics · Case Studies
Core Definition: Software engineering is an engineering discipline concerned with all aspects of software production — from early specification through to maintaining the system after use. It applies appropriate theories and methods within organizational and financial constraints.
Software Products & Costs
Generic Products
  • Stand-alone systems sold to any customer
  • Specification owned by the developer
  • Examples: PC software, CAD tools, dentist appointment systems
  • Developer decides when to change the product
Customized Products
  • Commissioned by a specific customer
  • Specification owned by the customer
  • Examples: embedded control systems, air traffic control, traffic monitoring
  • Customer decides on changes
Software Costs Key Fact: Software costs often dominate computer system costs. ~60% of costs are development, 40% are testing. For custom software, evolution costs often exceed development costs.
Essential Attributes of Good Software
Maintainability
Must evolve to meet changing customer needs — a critical attribute in a changing business environment.
Dependability & Security
Includes reliability, security, and safety. Should not cause physical or economic damage on failure.
Efficiency
No wasteful use of system resources (memory, processor cycles). Includes responsiveness and processing time.
Acceptability
Must be understandable, usable, and compatible with other systems used by its target users.
Software Process Activities
  • Specification — define what the system should do
  • Development — design and program the software
  • Validation — check it meets customer requirements
  • Evolution — modify to reflect changing needs
Software Engineering Ethics (ACM/IEEE Code)
1. Public

Act consistently with the public interest.

2. Client & Employer

Act in the best interests of client/employer, consistent with public interest.

3. Product

Ensure products meet the highest professional standards.

4. Judgment

Maintain integrity and independence in professional judgment.

5. Management

Promote an ethical approach to software management.

6. Profession

Advance integrity and reputation of the profession.

7. Colleagues

Be fair to and supportive of colleagues.

8. Self

Participate in lifelong learning and promote an ethical approach.

Three Case Studies Used Throughout the Book
Insulin Pump

Embedded safety-critical system. Controls blood sugar via micro-pump. Low sugar → coma/death; high sugar → organ damage. Hard real-time, safety-critical.

MHC-PMS

Mental Health Care Patient Management System. Centralized patient database. Key concerns: privacy (confidentiality) and safety (suicidal patients).

Wilderness Weather Station

Data collection system with remote sensors. Collects temperature, pressure, rainfall, wind. Includes station maintenance system via satellite.

CH 02
Software Processes
Process Models · Activities · Change · RUP
Definition: A software process is a structured set of activities required to develop a software system. All involve specification, design/implementation, validation, and evolution. A software process model is an abstract representation of a process from a particular perspective.
Three Core Process Models
Waterfall Model
  • Plan-driven — phases in sequence
  • Requirements → Design → Implementation → Testing → Operation
  • Hard to accommodate change mid-process
  • Best for: large systems at multiple sites with stable requirements
  • Problem: inflexible partitioning
Incremental Development
  • Specification, development, validation interleaved
  • System developed as a series of versions
  • Benefits: reduced cost of change, easier feedback, faster delivery
  • Problems: process not visible, structure degrades over time
  • Can be plan-driven or agile
Reuse-Oriented SE
  • System assembled from existing components or COTS
  • Stages: component analysis → requirements modification → design with reuse → integration
  • Now the standard approach for many business systems
  • Types: web services, component packages, stand-alone COTS
Process Activities in Detail
Requirements Engineering Process
  • Feasibility study — technically and financially feasible?
  • Requirements elicitation & analysis — what do stakeholders need?
  • Requirements specification — define requirements in detail
  • Requirements validation — check validity of requirements
Design Activities
  • Architectural design — overall structure and components
  • Interface design — interfaces between components
  • Component design — design of individual components
  • Database design — data structures and representations
Testing Stages
Stage 1
Development Testing
Individual components tested independently
Stage 2
System Testing
Test system as a whole; emergent properties
Stage 3
Acceptance Testing
Test with customer data; meets customer needs?
Coping With Change
Change Avoidance

Include activities that anticipate changes before rework is needed. E.g., build a prototype to explore requirements.

Change Tolerance

Design the process so changes can be made at relatively low cost — usually via incremental development.

Prototyping
A prototype is an initial version of a system used to demonstrate concepts and try out design options. Benefits: improved usability, closer match to needs, improved design quality, reduced development effort. Throw-away prototypes should be discarded — they are not a good basis for production.
Rational Unified Process (RUP)
Phase 1
Inception
Establish business case
Phase 2
Elaboration
Understand problem domain & architecture
Phase 3
Construction
System design, programming, testing
Phase 4
Transition
Deploy to operating environment
RUP Good Practice: Develop iteratively · Manage requirements · Use component-based architectures · Visually model software · Verify quality · Control changes.
Incremental Development & Delivery

Rather than deliver the system as a single delivery, the development and delivery is broken down into increments with each increment delivering part of the required functionality. User requirements are prioritised and the highest priority requirements are included in early increments. Once the development of an increment is started, the requirements are frozen though requirements for later increments can continue to evolve.

Incremental development and delivery
Incremental development

Develop the system in increments and evaluate each increment before proceeding to the development of the next increment; this is the normal approach used in agile methods. Evaluation is typically done by a user or customer proxy.

Incremental delivery

Deploy an increment for use by end-users so practical evaluation of the software is possible. This gives more realistic feedback about practical use, although it can be difficult to implement for replacement systems because early increments have less functionality than the system being replaced.

Incremental delivery advantages
Early customer value

Customer value can be delivered with each increment so system functionality is available earlier.

Prototyping for requirements

Early increments act as a prototype to help elicit requirements for later increments.

Lower risk

Lower risk of overall project failure; highest priority services tend to receive the most testing.

Incremental delivery problems

Most systems require a set of basic facilities that are used by different parts of the system. As requirements are not defined in detail until an increment is to be implemented, it can be hard to identify common facilities that are needed by all increments. The essence of iterative processes is that the specification is developed in conjunction with the software. However, this conflicts with the procurement model of many organisations, where the complete system specification is part of the system development contract.

Boehm's Spiral Model

The process is represented as a spiral rather than as a sequence of activities with backtracking. Each loop in the spiral represents a phase in the process. There are no fixed phases such as specification or design — loops in the spiral are chosen depending on what is required. Risks are explicitly assessed and resolved throughout the process.

Spiral model sectors
Objective setting

Specific objectives for the phase are identified.

Risk assessment & reduction

Risks are assessed and activities put in place to reduce the key risks.

Development & validation

A development model for the system is chosen which can be any of the generic models.

Planning

The project is reviewed and the next phase of the spiral is planned.

Spiral model usage

The spiral model has been very influential in helping people think about iteration in software processes and introducing the risk-driven approach to development. In practice, however, the model is rarely used exactly as published for practical software development.

CH 03
Agile Software Development
Agile Methods · XP · Scrum · Scaling
Agile Manifesto Core Values: Individuals & interactions over processes & tools · Working software over comprehensive documentation · Customer collaboration over contract negotiation · Responding to change over following a plan.
Agile Principles
Customer Involvement
Customers closely involved throughout; they provide and prioritize requirements and evaluate iterations.
Incremental Delivery
Software developed in increments; customer specifies requirements for each increment.
People Not Process
Team skills recognized and exploited; team develops its own ways of working.
Embrace Change
Expect requirements to change; design the system to accommodate them.
Maintain Simplicity
Focus on simplicity in the software and the development process. Actively eliminate complexity.
Extreme Programming (XP)
XP Practices (a)
  • Incremental planning — story cards; stories broken into tasks
  • Small releases — minimal useful set of functionality first
  • Simple design — just enough to meet current requirements
  • Test-first development — write tests before code
  • Refactoring — continuous code improvement
XP Practices (b)
  • Pair programming — developers work in pairs
  • Collective ownership — anyone can change anything
  • Continuous integration — integrate and test after every task
  • Sustainable pace — no excessive overtime
  • On-site customer — customer is part of the team
Scrum
Scrum Process
  • Phases: Outline planning → Sprint cycles → Project closure
  • Sprints are fixed length (2–4 weeks)
  • Starting point: product backlog
  • Team isolated during sprint — comms through Scrum master
  • Sprint ends with review/demo to stakeholders
Scrum Benefits
  • Product broken into manageable chunks
  • Unstable requirements don't hold up progress
  • Whole team has visibility — improved communication
  • Customers see on-time delivery of increments
  • Trust between customers and developers established
Scaling Agile
Scaling Up (Large Systems)
  • Need more up-front design and documentation
  • Cross-team communication mechanisms essential
  • Continuous integration is practically impossible at scale
  • Large systems often "brownfield" — interact with existing systems
Scaling Out (Large Companies)
  • Managers may resist risk of new approach
  • Quality procedures may conflict with agile
  • Wide range of skill levels in large organizations
  • Cultural resistance — long history with conventional processes
CH 04
Requirements Engineering
Functional · Non-Functional · Elicitation · Validation · Management
Definition: Requirements engineering is the process of establishing the services the customer requires from a system and the constraints under which it operates and is developed. Requirements are descriptions of system services and constraints generated during this process.
Types of Requirements
Functional Requirements
  • Statements of services the system should provide
  • How system reacts to particular inputs
  • What the system should NOT do
  • Example: "A user shall be able to search appointment lists for all clinics"
Non-Functional Requirements
  • Constraints on services or functions
  • Timing constraints, standards, dev-process constraints
  • Often apply to the system as a whole
  • May be more critical than functional — if not met, system unusable
Domain Requirements
  • Constraints from the operational domain
  • May be new functional requirements or restrictions
  • Problems: understandability, implicitness
  • If not satisfied, system may be unworkable
Non-Functional Requirement Classifications
Product Requirements

The delivered product must behave in a particular way — execution speed, reliability, etc.

Organisational Requirements

Consequence of organisational policies — process standards, implementation requirements.

External Requirements

From factors external to the system — interoperability, legislative requirements.

NFR Metrics
PropertyMeasure
SpeedProcessed transactions/second · User/event response time · Screen refresh time
SizeMbytes · Number of ROM chips
Ease of useTraining time · Number of help frames
ReliabilityMean time to failure · Probability of unavailability · Rate of failure occurrence
RobustnessTime to restart after failure · Percentage of events causing failure
PortabilityPercentage of target-dependent statements · Number of target systems
Requirements Elicitation Techniques
Interviews
  • Formal (pre-determined questions) or open/informal
  • Good for overall understanding of stakeholder work
  • Not good for understanding domain requirements
  • Must be open-minded; use springboard questions
Ethnography
  • Social scientist observes how people actually work
  • Reveals requirements from actual work vs. prescribed work
  • Good for understanding existing processes
  • Cannot identify new features that should be added
Scenarios / Use Cases
  • Real-life examples of system use
  • Include: starting situation, normal flow, what can go wrong, end state
  • Use cases identify actors and describe interactions
  • Sequence diagrams add detail to use cases
Requirements Validation
Key fact: Fixing a requirements error after delivery may cost up to 100× the cost of fixing an implementation error.
Validity

Does the system provide functions which best support customer needs?

Consistency

Are there any requirements conflicts or contradictions?

Completeness

Are all functions required by the customer included?

Realism

Can requirements be implemented given available budget and technology?

Verifiability

Can the requirements be tested/checked?

Requirements Change Management
Problem Analysis
Analyze if change is valid; fed back to requestor
Change Analysis
Assess effect using traceability; cost?
Change Implementation
Modify requirements doc, design, and implementation
CH 05
System Modeling
Context · Interaction · Structural · Behavioral · MDE
Definition: System modeling is developing abstract models of a system, each presenting a different view. Now almost always uses graphical notation based on UML. Models help analysts understand functionality and communicate with customers.
System Perspectives
External Perspective
Model the context or environment — what lies outside the system boundaries.
Interaction Perspective
Model interactions between system and environment, or between system components.
Structural Perspective
Model the organization of the system or data structures processed.
Behavioral Perspective
Model dynamic behavior and how the system responds to events.
UML Diagram Types
Activity Diagrams

Show activities involved in a process or data processing.

Use Case Diagrams

Show interactions between a system and its environment — actors and use cases.

Sequence Diagrams

Show interactions between actors and system, and between system components over time.

Class Diagrams

Show object classes in the system and associations between them.

State Diagrams

Show how the system reacts to internal and external events — states and transitions.

Behavioral Models
Data-Driven Modeling
  • Driven by data — controlled by input data
  • Shows sequence of actions in processing data
  • Useful for showing end-to-end processing
  • Represented using activity diagrams
Event-Driven / State Machine
  • System has finite number of states
  • Events cause transitions between states
  • Represented using UML statecharts
  • Used for real-time systems (e.g. microwave oven, insulin pump)
Model-Driven Engineering (MDE)
MDE: models rather than programs are the principal outputs. Programs are generated automatically from models. Pros: higher abstraction, cheaper platform adaptation. Cons: models may not be right for implementation; translation costs. MDA uses CIM → PIM → PSM transformation chain.
CH 06
Architectural Design
Patterns · Views · MVC · Layered · Repository · Client-Server · Pipe & Filter
Definition: Architectural design identifies the sub-systems making up a system and the framework for sub-system control and communication. It is an early stage, linking specification and design processes. Advantages: Stakeholder communication, system analysis, large-scale reuse.
4+1 View Model of Software Architecture
Logical View
Key abstractions as objects or object classes.
Process View
At run-time, how the system is composed of interacting processes.
Development View
How the software is decomposed for development.
Physical View
System hardware and how software is distributed across processors.
Architecture & System Characteristics
  • Performance — localise critical ops; use large-grain components
  • Security — layered architecture; critical assets in inner layers
  • Safety — localise safety-critical features
  • Availability — include redundant components, fault tolerance
  • Maintainability — fine-grain, replaceable components
Key Architectural Patterns
MVC (Model-View-Controller)
  • Separates presentation and interaction from system data
  • Model — manages data and operations
  • View — defines and manages data presentation
  • Controller — manages user interaction
  • When used: multiple ways to view/interact with data; unknown future requirements
  • Advantage: data and representation can change independently
  • Disadvantage: extra complexity when data model and interactions are simple
Layered Architecture
  • Organises system into layers with related functionality
  • Lower layers represent core services
  • When used: building on existing systems; multi-team development; multi-level security
  • Advantage: entire layers can be replaced if interface maintained
  • Disadvantage: clean separation is often difficult; performance issues
Repository Architecture
  • All data managed in a central repository; all components access it
  • Components do not interact directly — only through the repository
  • When used: large volumes of data must be stored for a long time
  • Advantage: components are independent; changes propagated to all; consistent data management
  • Disadvantage: single point of failure; inefficiencies in communication; hard to distribute
Client-Server Architecture
  • Stand-alone servers provide specific services (print, data management, etc.)
  • Clients call on these services
  • When used: shared database accessed from many locations; variable load
  • Advantage: servers can be distributed; general functionality available to all clients
  • Disadvantage: single point of failure; unpredictable performance; management problems if servers are in different organisations
Pipe and Filter Architecture
  • Each processing component (filter) is discrete and carries out one type of transformation
  • Data flows (as in a pipe) from one component to another
  • When used: data processing applications (batch and transaction-based)
  • Advantage: easy to understand; transformation reuse; can be sequential or concurrent
  • Disadvantage: data transfer format must be agreed; parsing/unparsing overhead

Design Patterns
A design pattern is a way of reusing abstract knowledge about a problem and its solution. Pattern elements: Name · Problem description · Solution description (template, not concrete design) · Consequences.
Observer Pattern
  • Separates display of object state from object itself
  • Used when multiple displays of state are needed
  • When state changes, all displays automatically notified
  • Consequence: optimisations to enhance display performance are impractical
Other Common Patterns
  • Façade — tidy interfaces to related objects
  • Iterator — standard way of accessing collection elements
  • Decorator — extend class functionality at run-time
Reuse Levels
Abstraction Level
Use knowledge of successful abstractions in your design — don't reuse software directly.
Object Level
Directly reuse objects from a library.
Component Level
Reuse collections of objects and classes as application components.
System Level
Reuse entire application systems.
Open Source Licenses

  • LGPL — can link to open source without publishing your components
  • BSD — non-reciprocal: can include in proprietary systems that are sold
CH 08
Software Testing
Development Testing · TDD · Release Testing · User Testing
Key Principle: Testing can reveal the presence of errors, NOT their absence. Goal 1: demonstrate software meets requirements (validation testing). Goal 2: discover situations where behaviour is incorrect (defect testing).
Verification vs Validation
Verification

"Are we building the product right?" — Software should conform to its specification.

Validation

"Are we building the right product?" — Software should do what the user really requires.

Stages of Testing
Development Testing
  • Unit testing — individual components in isolation
  • Component testing — groups of units integrated; test interfaces
  • System testing — all components integrated; test interactions
Release Testing
  • Separate team tests a complete version
  • Black-box testing — derived from specification only
  • Convince customer system is good enough
  • Includes performance/stress testing
User Testing
  • Alpha — users test at developer's site
  • Beta — release to users for experimentation
  • Acceptance — customer decides if system should be accepted
Testing Strategies
Partition Testing

Identify groups of inputs with common characteristics (equivalence partitions). Choose tests from each partition — the program behaves equivalently for all members of a class.

Guideline-Based Testing

Use testing guidelines based on previous experience of common programmer errors. For sequences: test single values, sequences of different sizes, access first/middle/last elements, test zero-length sequences.

Test-Driven Development (TDD)
Identify Increment
Small, implementable functionality
Write Test First
As automated test; initially fails
Implement
Write code to pass the test
Re-run All Tests
All must pass; refactor if needed
Next Increment
Move to next functionality
TDD Benefits: Code coverage (every segment has a test) · Regression testing (incremental test suite) · Simplified debugging (failure points to new code) · Documentation (tests describe what code should do).
Inspections vs Testing
Advantages of Inspections
  • Errors don't mask other errors (static)
  • Incomplete versions can be inspected
  • Can check broader quality attributes (standards, portability)
  • Can be used before implementation
Limitations of Inspections
  • Cannot check conformance with customer's real requirements
  • Cannot check non-functional characteristics (performance, usability)
  • Inspections and testing are complementary, not opposing
CH 09
Software Evolution
Evolution Processes · Lehman's Laws · Maintenance · Reengineering · Legacy Systems
Key Fact: The majority of the software budget in large companies is devoted to changing and evolving existing software rather than developing new software.
Evolution vs Servicing vs Phase-out
Evolution

In operational use; new requirements proposed and implemented. System is actively growing.

Servicing

Remains useful but only bug fixes and environment changes. No new functionality added.

Phase-out

Software may still be used but no further changes are made to it.

Types of Maintenance
Fault Repair

Changing system to correct deficiencies in how it meets requirements.

Adaptive Maintenance

Changing system to operate in a different environment (hardware, OS, etc.).

Perfective Maintenance

Modifying system to satisfy new or changed requirements.

Lehman's Laws
LawDescription
Continuing ChangeA program in real-world use must change or become progressively less useful.
Increasing ComplexityAs a program evolves, its structure becomes more complex; extra resources needed to preserve simplicity.
Large Program EvolutionProgram evolution is self-regulating — size, release intervals, errors are approximately invariant per release.
Organizational StabilityDevelopment rate approximately constant over lifetime, independent of resources devoted.
Conservation of FamiliarityIncremental change in each release is approximately constant over lifetime.
Continuing GrowthFunctionality must continually increase to maintain user satisfaction.
Declining QualityQuality will decline unless modified to reflect changes in operational environment.
Feedback SystemEvolution processes are multiagent, multiloop feedback systems.
Reengineering vs Refactoring
Reengineering
  • Re-structuring or re-writing part/all of a legacy system without changing functionality
  • After a system has been maintained for some time
  • Uses automated tools
  • Advantages: reduced risk, reduced cost vs new development
Refactoring
  • Continuous improvement throughout development/evolution
  • "Preventative maintenance" — slows down degradation
  • Improves structure, reduces complexity; does NOT add functionality
  • "Bad smells": duplicate code, long methods, switch statements, data clumping, speculative generality
Legacy System Strategy
QualityBusiness ValueStrategy
LowLowScrap the system
LowHighRe-engineer or replace
HighLowReplace with COTS or scrap
HighHighContinue with normal maintenance
CH 10
Sociotechnical Systems
Complex Systems · Emergent Properties · Systems Engineering · Procurement · Operation
Sociotechnical Systems include technical systems plus operational processes and people who use them. They are governed by organisational policies and rules. They are non-deterministic and have complex relationships with organisational objectives.
STS Stack (Layers)
Equipment
Hardware devices (usually include an embedded system).
Operating System
Provides common facilities for higher levels.
Communications & Data Management
Middleware for remote systems and databases.
Application Systems
Specific functionality for organisational requirements.
Business Processes
Processes involving people and computer systems supporting business activities.
Organisations → Society
Higher-level strategic activities; laws, regulation, culture.
Emergent Properties
Properties of the system as a whole that cannot be derived from the properties of components. Can only be assessed once components are integrated. Examples: Volume, Reliability, Security, Repairability, Usability. Two types: Functional (appear when all parts work together) and Non-functional (reliability, performance, safety, security).
Human Error: Swiss Cheese Model
Failures in complex systems rarely have a single cause. Systems are designed with multiple barriers (diverse and redundant). No single barrier is perfect — there will be latent conditions. With multiple barriers, all must fail for a system failure to occur. Reason's Swiss Cheese model: holes in layers of defence must align for an accident to occur.
Systems Engineering Stages
Procurement
Establish purpose; high-level requirements; decide on components to purchase
Development
Define requirements in detail; implement and test; define operational processes
Operation
Deploy; put into use; make changes as requirements emerge; decommission
CH 11
Security & Dependability
Dependability Properties · Availability · Reliability · Safety · Security
Dependability reflects the user's degree of trust in the system — confidence that it will operate as expected and not fail in normal use. Covers reliability, availability, and security, which are inter-dependent.
Principal Dependability Properties
Availability
Probability that the system will be up and running and able to deliver useful services. Takes repair time into account.
Reliability
Probability the system will correctly deliver services as expected. Defined formally against a specification.
Safety
How likely it is that the system will cause damage to people or the environment. Safety and reliability are related but distinct.
Security
How likely the system can resist accidental or deliberate intrusions. Essential pre-requisite for availability and reliability.
Repairability
Extent to which system can be repaired after failure. Short-term perspective.
Maintainability
Ease of adapting system to new requirements. Long-term perspective.
Survivability
Ability to continue delivering services under hostile attack. Subsumes resilience.
Error Tolerance
Extent to which user errors can be avoided and tolerated.
Reliability Terminology
TermDefinition
Human Error/MistakeHuman behaviour that introduces faults into a system.
System FaultCharacteristic of software that can lead to a system error.
System ErrorErroneous system state that can lead to unexpected behaviour.
System FailureEvent when system does not deliver service as expected by users.
Key Insight: Removing X% of faults does NOT necessarily improve reliability by X%. Faults in rarely-executed code may never be encountered. A program with known faults may still be perceived as reliable by users.
Security Terminology
TermDefinition
AssetSomething of value to be protected (software or data).
ExposurePossible loss or harm — loss/damage to data, or loss of time/effort.
VulnerabilityWeakness in a system that may be exploited to cause loss or harm.
AttackExploitation of a system vulnerability — usually deliberate, from outside.
ThreatCircumstances that have potential to cause loss or harm (vulnerability + attack).
ControlProtective measure that reduces a system's vulnerability (e.g. encryption).
CH 12
Dependability & Security Specification
Risk-Driven Spec · Safety Spec · Reliability Metrics · Security Spec · Formal Methods
Risk-Driven Specification Stages
Risk Identification
Identify potential risks
Risk Analysis & Classification
Assess seriousness of each risk
Risk Decomposition
Find root causes of risks
Risk Reduction Assessment
Define how each risk must be eliminated/reduced
Hazard Assessment
Intolerable Risk

Must never arise or result in an accident. No exceptions.

ALARP

As Low As Reasonably Practical. Minimise possibility given cost and schedule constraints.

Acceptable Risk

Consequences are acceptable; no extra costs should be incurred to reduce hazard probability.

Fault-Tree Analysis — deductive, top-down technique. Put the hazard at the root; identify system states that could lead to it. Link with 'and'/'or' conditions. Goal: minimise number of single causes of failure.
Reliability Metrics
POFOD

Probability of Failure on Demand. For intermittent service demands. E.g. emergency shutdown systems. POFOD of 0.002 = 1 failure in 500 demands.

ROCOF

Rate of Occurrence of Failures. For high-volume processing. ROCOF 0.002 = 2 failures per 1000 hours. Reciprocal = MTTF (Mean Time to Failure).

Availability

Fraction of time system is available. Takes repair time into account. Availability 0.999 = unavailable for 84 seconds per 24-hour period.

Formal Methods
Formal methods use mathematical representation and analysis. Include: formal specification, specification analysis and proof, transformational development, program verification. Pros: forces detailed analysis; concurrent systems can be checked; detects implementation errors. Cons: specialised notation; very expensive; proofs may contain errors; hard to scale.
CH 13
Dependability Engineering
Redundancy · Diversity · Dependable Architectures · N-Version Programming · Dependable Programming
Dependability Achievement Approaches
Fault Avoidance

Develop in a way that minimises the possibility of mistakes and prevents faults being introduced.

Fault Detection

V&V techniques to discover and remove faults before deployment.

Fault Tolerance

System continues in operation despite software faults in delivered software.

Redundancy & Diversity
Redundancy

Keep more than one version of a critical component. If one fails, a backup is available. E.g., backup servers in e-commerce systems.

Diversity

Provide the same functionality in different ways so they will not fail in the same way. E.g., different OS versions (Windows + Linux) for resilience against external attacks.

Dependable System Architectures
Protection Systems
  • Specialised system associated with a control system to take emergency action on failure
  • Independently monitors the controlled system and environment
  • Redundant: includes monitoring and control capabilities that replicate the control software
  • Diverse: uses different technology from the control software
  • Simpler than control system — more effort can be spent on validation
Self-Monitoring Architectures
  • Multi-channel architecture; system monitors its own operations
  • Same computation carried out on each channel; results compared
  • If results differ, failure is assumed and exception raised
  • Hardware and software in each channel must be diverse
  • Example: Airbus flight control system (5 computers, diverse processors, diverse OS, diverse programming languages)
N-Version Programming
  • Multiple (odd number, typically 3) versions carry out computations simultaneously
  • Results compared using a voting system; majority result taken as correct
  • Different teams develop each version; assumed low probability of same mistakes
  • Problem: teams often misinterpret specifications in the same way; chose same algorithms
  • Reliability improvement of 5–9× observed in practice
Dependable Programming Guidelines
8 Key Guidelines
  • Limit the visibility of information in a program
  • Check all inputs for validity
  • Provide a handler for all exceptions
  • Minimize the use of error-prone constructs
  • Provide restart capabilities
  • Check array bounds
  • Include timeouts when calling external components
  • Name all constants that represent real-world values
Error-Prone Constructs to Avoid
  • Unconditional goto statements
  • Floating-point numbers (imprecise comparisons)
  • Pointers (data corruption, aliasing)
  • Dynamic memory allocation (overflow)
  • Parallelism (timing errors)
  • Recursion (stack overflow)
  • Inheritance (non-localised code)
  • Unbounded arrays (buffer overflow)
CH 14
Security Engineering
Risk Management · Architectural Design · Design Guidelines · Survivability
Security Engineering: Tools, techniques and methods to support development and maintenance of systems that can resist malicious attacks. Application security = software engineering problem (design to resist attacks). Infrastructure security = systems management problem (configure infrastructure).
Misuse Cases (Threat Types)
Interception
Attacker gains access to an asset (loss of confidentiality).
Interruption
Attacker makes part of a system unavailable (denial of service).
Modification
A system asset is tampered with.
Fabrication
False information is added to a system.
Design Guidelines for Secure Systems
Guidelines 1–5
  • Base decisions on explicit security policy
  • Avoid a single point of failure — failure only if more than one security failure
  • Fail securely — sensitive info not accessible even on failure
  • Balance security and usability — weaker security may sometimes be needed
  • Log user actions — analysis of who did what; deterrent
Guidelines 6–10
  • Use redundancy and diversity to reduce risk
  • Validate all inputs — check all inputs are in range
  • Compartmentalize your assets — users only access what they need
  • Design for deployment — avoid deployment problems
  • Design for recoverability — simplify recovery after attack
Survivability Strategies
Resistance

Build capabilities into the system to resist attacks — avoid problems.

Recognition

Build capabilities to detect attacks and failures, and assess the damage.

Recovery

Build capabilities to deliver services whilst under attack — tolerate problems.

Static Analysis Techniques
Formal Verification

Mathematical specification analyzed for consistency; program conformance to spec formally argued. Discovers spec errors, design errors, concurrency issues.

Model Checking

Extended finite state model created; model checker explores all possible paths; checks user-specified property is valid on each path. Valuable for concurrent systems.

Automated Static Analysis

Software tools parse program text and find potentially erroneous conditions. Very effective as aid to inspections. Particularly valuable for C (weak typing) and security checking (buffer overflows).

Reliability Testing

Establish statistical model of use
Build Test Data
Reflects operational profile
Test System
Observe failures and times
Compute Reliability
After statistically significant failures
Safety & Dependability Cases
A safety case is "a documented body of evidence that provides a convincing and valid argument that a system is adequately safe for a given application in a given environment." Required by regulators before a system can be certified. Based on structured arguments that use claims, evidence, and arguments to demonstrate safety.
Structured Safety Arguments: Based on proof by contradiction — assume unsafe state can be reached; show this is contradicted by the program code. Start from END of code, work backwards. Show all paths leading to exit contradict the assumption of unsafe exit.
CH 16
Software Reuse
Reuse Landscape · Application Frameworks · Product Lines · COTS · ERP
Benefits of Reuse: Increased dependability (tried and tested) · Reduced process risk · Effective use of specialists · Standards compliance · Accelerated development. Problems: Increased maintenance costs · Lack of tool support · Not-invented-here syndrome · Component library management costs.
Reuse Approaches
Application Frameworks
  • Sub-system design made up of abstract and concrete classes
  • Types: System infrastructure, Middleware integration, Enterprise application, Web application (WAF)
  • MVC is a system infrastructure framework for GUI design
  • WAF features: security, dynamic web pages, database support, session management, AJAX
  • Problem: high complexity — takes time to learn to use effectively
COTS Products
  • Systems adapted for different customers without changing source code
  • Benefits: rapid deployment, visible functionality, reduced development risk
  • Problems: requirements must be adapted; COTS product vendor controls evolution
  • COTS-solution: single product, standard processes, vendor maintains
  • COTS-integrated: multiple heterogeneous systems, flexible, customer maintains
Software Product Lines
A set of applications with a common architecture and shared components, each specialized to reflect different requirements. Specialization types: Platform, Environment, Functional, Process. Configuration types: Design time (source code modified) and Deployment time (embedded knowledge; source not changed).
ERP Systems
Enterprise Resource Planning systems support common business processes (ordering, invoicing, manufacturing). The most common form of software reuse in large companies. Configured via: selecting modules, establishing data model, defining business rules, specifying external interactions, designing forms/reports, setting parameters.
CH 17
Component-Based Software Engineering
Components · Component Models · CBSE Processes · Composition
Component Definition (Szyperski): A software component is a unit of composition with contractually specified interfaces and explicit context dependencies only. It can be deployed independently and is subject to composition by third-parties. Key distinction from objects: components have no 'requires' interface dependency on specific components.
Component Characteristics
Standardized
Conforms to a standard component model (interfaces, metadata, composition, deployment).
Independent
Can be composed and deployed without other specific components; external services explicitly stated.
Composable
All external interactions through publicly defined interfaces; external access to metadata.
Deployable
Self-contained; binary (no compilation needed); can operate standalone on a component platform.
Documented
Fully documented so potential users can decide if component meets their needs.
Component Interfaces
Provides Interface

Services that the component provides to others — the component's API. Methods that can be called by users of the component.

Requires Interface

Services that must be made available for the component to execute as specified. Does NOT compromise independence — does not define HOW services should be provided.

Types of Component Composition
Sequential

Composed components executed in sequence. Composes the provides interfaces of each component.

Hierarchical

One component calls on services of another. Provides interface of one composed with requires interface of another.

Additive

Interfaces of two components put together to create a new component with combined interfaces.

Interface Incompatibility Types: Parameter incompatibility (same name, different types) · Operation incompatibility (different names for same operations) · Operation incompleteness (provides interface is a subset of requires). Resolved by Adaptor Components.

Distributed System Design Issues
Transparency
To what extent should the distributed system appear to users as a single system? Resources abstracted logically; middleware maps logical to physical.
Openness

CH 20 / RT
Embedded Systems & Real-Time SE
Hard/Soft Real-Time · Architectural Patterns · Timing Analysis · RTOS
Real-Time System: Correct functioning depends on the results produced AND the time at which these results are produced. Hard real-time = operation is incorrect if timing not met. Soft real-time = operation is degraded if timing not met.
Hard vs Soft Real-Time Examples
Hard Real-Time
  • Vehicle braking system
  • Insulin monitoring and delivery system
  • Microwave oven
  • Missile guidance system
Soft Real-Time
  • Wilderness weather station
  • Video streaming systems
  • User interface response
Architectural Patterns for Embedded Systems
Observe & React

Routinely monitor sensors; display values; if exceptional condition → alarm/action. Used in monitoring and alarm systems.

Environmental Control

Analyze sensor data + actuator state; send control signals to actuators to change environment. Used in control systems.

Process Pipeline

Data moves in sequence from one end to another through processes linked by synchronized buffers. Used in data acquisition and multimedia systems.

Timing Analysis
Three key factors: Deadlines (times by which stimuli must be processed), Frequency (times per second a process must execute), Execution time (time required to process stimulus). Timing results determine process scheduling in the RTOS.
RTOS Components
Real-time Clock

Provides information for process scheduling.

Interrupt Handler

Manages aperiodic requests for service.

Scheduler

Chooses the next process to be run.

Resource Manager

Allocates memory and processor resources.

Dispatcher

Starts process execution.

Scheduling Strategies
Non Pre-emptive

Once scheduled, process runs to completion (or until blocked for I/O).

Pre-emptive

Execution of a process can be stopped if a higher-priority process requires service.

Algorithms

Round-robin · Rate monotonic · Shortest deadline first

Aspect-Oriented Software Engineering
Separation of Concerns · Aspects · Join Points · Pointcuts · Weaving
Cross-cutting concerns are concerns whose implementation cuts across program components, leading to tangling (one module implements different requirements) and scattering (implementation of one concern scattered across many modules). Aspects solve this by encapsulating cross-cutting concerns.
Key AOSE Terminology
TermDefinition
AspectProgram abstraction that defines a cross-cutting concern. Includes pointcut definition and associated advice.
AdviceThe code implementing a concern.
Join PointEvent in an executing program where aspect advice may be executed.
PointcutStatement defining the join points where the aspect advice should be executed.
WeavingIncorporation of advice code at the specified join points by an aspect weaver.
AspectJ Join Point Model Events
Call Events Execution Events Initialisation Events Data Events Exception Events
V&V Problems with AOSE: Cannot read programs top-to-bottom (they are a web). Cannot derive traditional program flow graphs. Test coverage is unclear (each aspect once? At each join point?). Difficult to design systematic tests. Teams prefer conventional approaches for large projects.

CH 25
Configuration Management
Change Management · Version Management · System Building · Release Management
CM Definition: Concerned with the policies, processes and tools for managing changing software systems. Needed because it is easy to lose track of what changes and component versions have been incorporated into each system version.
CM Key Terminology
Configuration Item (SCI)
Anything associated with a software project placed under configuration control — has a unique name and may exist in multiple versions.
Version
An instance of a configuration item that differs from other instances. Identified by name + version number.
Baseline
A collection of component versions making up a system. Controlled — component versions cannot be changed. System can always be recreated.
Codeline
A set of versions of a software component and its dependencies — shows evolution history of a component.
Mainline
A sequence of baselines representing different versions of a system.
Branch / Merge
Branch: new codeline from an existing version. Merge: new version created by combining separate codeline versions.
Git (Distributed VCS)
Created by Linus Torvalds in 2005 to manage the Linux kernel. Distributed — maintains a clone of the repository on every user's computer. Benefits: Resilience (everyone has a copy), Speed (local operations), Flexibility (safe local experimentation). Mainstream since ~2015 with Docker containers for DevOps.
System Building
Development System

Development tools (compilers, editors). Developers check out code into private workspace.

Build Server

Used to build definitive executable versions. Developers check in code before building. May use external libraries not in VCS.

Target Environment

Platform on which system executes. May be different hardware/OS from development system.

Minimizing Recompilation: Tools compare modification timestamps (or checksums) of source vs object code. Only recompile when source code has changed. Checksums allow parallel compilation; timestamps may not. "Breaking the build" = pushing code that causes system tests to fail. Use "integrate twice" approach — test locally first.
